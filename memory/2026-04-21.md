- 2026-04-21: No projeto Mutation AI Studio, o fluxo de `mutation-ai create test` foi consolidado como: gerar prompts por classe selecionada, salvar em `.mutation-ai/prompts/create-test-<timestamp>/`, gerar respostas do Ollama e salvar em `.mutation-ai/generated/...`, sanitizar e validar, salvar apenas aprovados em `.mutation-ai/approved/...`; promoção para `src/test/java` ficou adiada para um comando futuro separado.
- 2026-04-21: A validação foi confirmada como parte real do pipeline antes da aprovação. `CreateTestCliAdapter` chama `create -> generate -> validate`, e arquivos ruins podem existir em `generated`, mas só devem chegar a `approved` se passarem na validação.
- 2026-04-21: A UX de aprovados foi ajustada para sempre garantir a criação da pasta `.mutation-ai/approved/create-test-<timestamp>/`, mesmo quando o lote tem zero aprovados, mantendo a pasta vazia e exibindo no console a mensagem: `Nenhum teste foi aprovado neste lote; nada foi salvo em approved`.
- 2026-04-21: O template de prompt em `CreateTestPromptService` foi generalizado para qualquer projeto Java, removendo referências específicas como `LoginService`. As regras agora são baseadas em padrões de código: usar `assertNull` apenas quando o fluxo real explicitamente retornar `null`; nunca usar `assertNull` em fluxos com `Optional.get()`, `orElseThrow()` ou dereferência posterior; em cenários not found com fluxo de exceção usar `assertThrows`; para métodos `void` não capturar retorno e validar com `verify(...)`; quando houver delegação para repositórios/serviços/clientes, verificar interações relevantes com `verify(...)`; evitar `assertNotNull` isolado quando houver interação relevante ou valor determinístico.
- 2026-04-21: Foi iniciado/implementado um ciclo automático de refinamento para testes rejeitados, com arquitetura hexagonal e use case separado (`RefineGeneratedTestsUseCase` / `RefineGeneratedTestsService`), limite máximo de 3 tentativas por classe, sem loop infinito. Cada tentativa usa um prompt de correção contendo o teste rejeitado anterior, os motivos da rejeição e instruções objetivas de correção.
- 2026-04-21: O refinamento deve preservar histórico e não sobrescrever versões anteriores. Cada tentativa deve ser salva separadamente com nomes claros por tentativa (ex.: `attempt1`, `attempt2`) e o CLI deve mostrar evolução por classe: número de tentativas, motivos que causaram refinamento, e status final aprovado/rejeitado.
